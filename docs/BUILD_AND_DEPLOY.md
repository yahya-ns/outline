# Build and Deploy

Outline is built once and run anywhere: a single `yarn build` produces a tree under `build/` that runs as a Node process, fits inside an official Docker image, or deploys to Heroku with one click. This document covers the build pipeline, runtime entry point, container images, deployment targets, migrations, the env vars that matter, the anonymous version ping, and the release process.

## Prerequisites

This doc assumes a working `yarn` install (Yarn 4, PnP) and familiarity with Node.js. To produce the official image you need Docker. To deploy to Heroku you need the Heroku CLI. To self-host on a Linux box you need systemd (or another supervisor), a reachable PostgreSQL 14+ instance, and a reachable Redis instance. For local development only, see `docs/DEVELOPMENT.md`.

## Build pipeline

`yarn build` runs four phases in order. The full script list lives in `package.json`.

1. `yarn clean` — removes the previous `build/` tree.
2. `yarn vite:build` — bundles the React app into `build/app/` using `rolldown-vite` with `@vitejs/plugin-react-oxc`. The build emits a PWA service worker and `manifest.webmanifest` via `vite-plugin-pwa`. Two Workbox runtime caches are configured: `unfurl-cache` (link previews) and `files-cache` (attachment URLs). Sourcemaps are emitted. `assetsInlineLimit: 0` keeps all assets external so a strict CSP can allow them by hash. `webpack-stats.json` is written for RelativeCI to consume. The `base` path honours `CDN_URL` so assets are served from `<CDN_URL>/static/...`. The vendor-chunk split is detailed in [What `yarn build` produces](#what-yarn-build-produces).
3. `yarn build:i18n` — runs `i18next-parser` against `app/`, `server/`, `shared/`, and `plugins/` to extract keys into `shared/i18n/locales/en_US/translation.json`, then `yarn copy:i18n` mirrors locale files into `build/shared/i18n/locales/`.
4. `yarn build:server` — invokes `build.js`. It wipes `build/server/` and `build/plugins/`, then Babel-compiles `server/`, `shared/`, and each plugin's `server/` and `shared/` directories into `build/server/`, `build/shared/`, and `build/plugins/<plugin>/`. It copies `server/collaboration/Procfile`, `server/static/error.{dev,prod}.html`, the root `package.json`, and each plugin's `plugin.json`.

The resulting layout is:

```
build
├── app                   # Static frontend (vite output + PWA assets)
├── server                # Compiled server entry: node build/server/index.js
├── shared                # Compiled shared utilities and editor
├── plugins/<plugin>      # Compiled plugin sources (one folder per plugin)
└── shared/i18n/locales   # Translation files served at /locales
```

### What `yarn build` produces

The four phases emit a tree under `build/`. The per-folder detail below extends the summary above.

#### `build/app/`

The vite bundle emits:

- `index.html` and hashed asset filenames following `assets/[name].[hash].js` and `assets/[name].[hash][extname]`.
- `assets/` — JS and CSS chunks, all content-hashed.
- `sw.js` — the Workbox-backed service worker generated by `vite-plugin-pwa`.
- `manifest.webmanifest` — the PWA manifest (name, icons, theme color).
- `webpack-stats.json` — emitted by `rollup-plugin-webpack-stats`, consumed by the bundle-size CI job.

Dependencies split into a fixed set of vendor chunks via `rollupOptions.output.advancedChunks.groups` in `vite.config.ts`:

| Chunk | Source packages |
| --- | --- |
| `vendor-shared` | `uuid`, `vite/preload-helper` |
| `vendor-react` | `react`, `react-dom`, `scheduler`, `react-router` |
| `vendor-prosemirror` | `prosemirror*`, `@benrbray/prosemirror*` |
| `vendor-collab` | `yjs`, `y-prosemirror`, `y-indexeddb`, `@hocuspocus/*`, `lib0` |
| `vendor-framer-motion` | `framer-motion` |
| `vendor-styled` | `styled-components` |
| `vendor-mermaid-elk` | `@mermaid-js/layout-elk`, `elkjs` |
| `vendor-mermaid` | `mermaid`, `cytoscape`, `cytoscape-fcose`, `layout-base`, `dagre-d3-es`, `langium`, `chevrotain`, `roughjs`, `@mermaid-js/*` |
| `vendor-katex` | `katex` |
| `vendor-emoji` | `@emoji-mart/*`, `emoji-mart` |
| `vendor-es-toolkit` | `es-toolkit` |
| `vendor-date` | `date-fns` |
| `vendor-sentry` | `@sentry/*` |

#### `build/server/`

Babel-compiled output of `server/` (the runtime entry is `node build/server/index.js`). The build also copies `server/collaboration/Procfile`, `server/static/error.dev.html`, `server/static/error.prod.html`, and the root `package.json` into the same tree.

#### `build/plugins/<plugin>/`

For each plugin in `plugins/` with a `server/` or `shared/` directory, Babel-compiled output. Each plugin's `plugin.json` is copied to `build/plugins/<plugin>/plugin.json`.

#### `build/shared/`

Babel-compiled `shared/` (utilities, editor components, theme tokens, i18n language metadata).

#### `build/shared/i18n/locales/`

Translated locale files mirrored by `yarn copy:i18n` from `shared/i18n/locales/`. Served at `/locales/<lang>.json` by the runtime.

A Babel config in `.babelrc` applies the same path aliases as runtime (`@server`, `@shared`, `~`, `plugins/*`), enables `transform-inline-environment-variables` for `SOURCE_COMMIT` and `SOURCE_VERSION` (populated from git during the release commit and embedded into the bundle for runtime introspection), and runs the `styled-components` displayName plugin in production.

### Static assets and CDN

`vite.config.ts` sets `publicDir` to `server/static/` (containing the error pages and favicons), and `base` is conditional on `CDN_URL`: when set, every asset URL is rewritten to `<CDN_URL>/static/...` so a CDN can serve JS, CSS, fonts, and images. Set the CDN origin to the same value as `URL` so relative paths resolve. The `assetsInlineLimit: 0` option forces all assets to be emitted as external files — required by the strict CSP `script-src` hash allowlist, which cannot hash inline `<script>` payloads reliably across builds. Partner and embed logos live under `public/` and are copied verbatim into `build/app/` by Vite.

## Behind a reverse proxy

When Outline sits behind nginx, Cloudflare, an AWS ALB, or Heroku's router, set `URL` to the public origin users hit, and `FORCE_HTTPS=true` to redirect HTTP to HTTPS at the app level (handled by `enforceHttps` from `koa-sslify` in `server/services/web.ts`). Set `PROXY_HEADERS_TRUSTED=true` so the app trusts the `X-Forwarded-*` headers — this is required for correct client IPs, the original protocol, and OAuth callback handling. Override `PROXY_IP_HEADER` (defaults to `X-Forwarded-For`) if your proxy uses a different header. When SSL terminates at the load balancer, set `FORCE_HTTPS=false` to skip the app-level redirect. The collaboration service can also be split off onto its own domain — see `COLLABORATION_URL` below.

## File storage backends

`FILE_STORAGE` selects the storage backend. `local` uses `server/storage/files/LocalStorage.ts` with the path from `FILE_STORAGE_LOCAL_ROOT_DIR` (defaults to `/var/lib/outline/data`, declared as a `VOLUME` in the Docker image). `s3` uses `S3Storage.ts` and reads the `AWS_*` family of variables; attachments and avatars go through the bucket named by `AWS_S3_UPLOAD_BUCKET_NAME` and the URL in `AWS_S3_UPLOAD_BUCKET_URL`. Optional CloudFront signing (`AWS_CLOUDFRONT_URL`, `AWS_CLOUDFRONT_KEY_PAIR_ID`, `AWS_CLOUDFRONT_PRIVATE_KEY`) accelerates download paths while uploads stay on S3. Size guards (`FILE_STORAGE_UPLOAD_MAX_SIZE`, `FILE_STORAGE_IMPORT_MAX_SIZE`, `FILE_STORAGE_WORKSPACE_IMPORT_MAX_SIZE`) cap uploads at the storage layer.

## Runtime

The compiled entry point is `node build/server/index.js` (the `yarn start` script). The process loads the service registry from `server/services/`, runs migrations and the telemetry ping once on the master, then forks one worker per CPU per service via `throng`.

Select the services for a given process with either the `--services` CSV flag or the `SERVICES` env var:

```bash
yarn start --services=web,worker
# or
SERVICES=collaboration yarn start
```

The available services (`web`, `worker`, `collaboration`, `websockets`, `admin`, `cron`) and their responsibilities are documented in `docs/SERVICES.md`.

## Docker

Two files drive container builds: `Dockerfile.base` and `Dockerfile`.

`Dockerfile.base` is the builder stage. It starts from `node:24.16.0`, installs `cmake`, enables Corepack, runs `yarn install --immutable`, copies the source, executes `yarn build`, and finally trims dev dependencies with `yarn workspaces focus --production`. The published image is `outlinewiki/outline-base`.

`Dockerfile` is the runner. It starts from `node:24.16.0-slim`, creates a non-root `nodejs` user (uid 1001), copies `build/`, `server/`, `public/`, `.sequelizerc`, `node_modules/`, and `package.json` from the builder stage, installs `wget` for the healthcheck, declares `/var/lib/outline/data` as a volume, and defines:

- `ENV NODE_ENV=production`
- `ENV FILE_STORAGE_LOCAL_ROOT_DIR=/var/lib/outline/data`
- `HEALTHCHECK --interval=1m CMD wget -qO- "http://localhost:${PORT:-3000}/_health" | grep -q "OK" || exit 1`
- `CMD ["node", "build/server/index.js"]`

The published image is `outlinewiki/outline`. Multi-arch (`amd64` + `arm64`) manifest lists are built and pushed by `.github/workflows/docker.yml` on tag pushes.

`docker-compose.yml` at the repo root brings up a local Redis and Postgres on `127.0.0.1` for development; the application itself is not in the compose file. See `docs/DEVELOPMENT.md` for the local workflow.

## Heroku

Heroku deploys work out of the box. `Procfile` defines two process types: `web` runs `yarn start --services=web,websockets,collaboration`, and `worker` runs `yarn start --services=worker`. `app.json` declares the addons (`heroku-redis`, `heroku-postgresql`), the required env (`NODE_ENV`, `SECRET_KEY`, `UTILS_SECRET`, `ENABLE_UPDATES`), and runs `yarn sequelize db:migrate` on deploy and review-app creation. The `heroku-postbuild` script runs `yarn build` before release.

## Migrations

Migrations live in `server/migrations/` and follow the sequelize-cli convention (`<TIMESTAMP>-<slug>.js`, CommonJS, exported `up`/`down`). `.sequelizerc` loads `.env` or `.env.test` via dotenvx and points sequelize-cli at `server/config/database.js` (env-driven via `DATABASE_URL`), `server/migrations/`, and `server/models/`. The Postgres dialect is configured with `dialectOptions.ssl.rejectUnauthorized=false` in production. For test runs, `.env.test` overrides `DATABASE_URL`, `REDIS_URL`, and `FILE_STORAGE` (to `local`) so unit tests do not need a real S3 bucket.

At runtime, the master process auto-runs pending migrations on boot through Umzug (wired in `server/storage/database.ts`). Pass `--no-migrate` to `yarn start` to short-circuit the auto-run (useful for read-only replicas or one-off migrations run by a separate process). The helper scripts are `yarn db:create-migration`, `yarn db:create`, `yarn db:migrate`, `yarn db:rollback`, and `yarn db:reset`.

## Self-hosting on a Linux box

For a bare-metal or VM install, run the build artifact under a process supervisor. The official container gives a useful reference process model: `node build/server/index.js` as a single supervised service, healthchecked on `/_health` over HTTP, with `/var/lib/outline/data` as the only persistent volume when `FILE_STORAGE=local`. The Postgres database and the Redis instance are separate stateful services that must be backed up independently — attachments and avatars live in S3 when configured to, otherwise on the local volume.

Tune `WEB_CONCURRENCY` (the `throng` worker count) by dividing available memory by ~512 MB. Cap database connections with `DATABASE_CONNECTION_POOL_MIN`/`DATABASE_CONNECTION_POOL_MAX` so the per-process pool does not exceed the Postgres `max_connections` limit. Logs go to stdout; pipe them to your aggregator of choice.

## Environment variables

`.env.sample` is the source of truth for every variable. The `_FILE` convention lets any variable load from a file (Docker secrets): set `SECRET_KEY_FILE=/run/secrets/...` and the loader reads and trims it; the direct variable takes precedence if both are set.

The variables below are the ones to verify first when bringing up an install. Defaults and additional vars live in `.env.sample`.

| Variable | Purpose |
| --- | --- |
| `URL` | Public origin used to build absolute links and OAuth callbacks. Must match what users hit in the browser (or the upstream proxy). |
| `DATABASE_URL` | Postgres connection string. Sequenced through `.sequelizerc` and the storage layer; `dialectOptions.ssl` is forced in production. |
| `REDIS_URL` | Primary Redis used for Bull queues, Socket.IO adapter, and rate limiting. |
| `SECRET_KEY` | 64 hex chars (`openssl rand -hex 32`). Symmetric key for `@Encrypted` model fields (API keys, OAuth tokens, attachment keys). |
| `UTILS_SECRET` | Independent random secret guarding the cron endpoint. |
| `FILE_STORAGE` | `local` or `s3`. Selects `server/storage/files/LocalStorage` or `S3Storage`. |
| `AWS_ACCESS_KEY_ID` | Required when `FILE_STORAGE=s3`. |
| `AWS_SECRET_ACCESS_KEY` | Required when `FILE_STORAGE=s3`. |
| `AWS_REGION` | Region of the upload bucket. |
| `AWS_S3_UPLOAD_BUCKET_NAME` | Bucket for attachments and avatars. |
| `AWS_S3_UPLOAD_BUCKET_URL` | Public URL the browser uses to PUT/GET objects (may be a CDN in front of S3). |
| `COLLABORATION_URL` | Optional `wss://...` URL pointing at a separately hosted collaboration service. See `docs/SERVICES.md`. |
| `RATE_LIMITER_ENABLED` | Master switch for the per-route and global rate limiters. Disabled in `.env.test`. |
| `LOG_LEVEL` | Winston threshold (`error`, `warn`, `info`, `http`, `verbose`, `debug`, `silly`). |
| `SENTRY_DSN` | Optional Sentry DSN for error reporting. MCP traffic is tagged `outline-mcp`. |

`FORCE_HTTPS` (defaults to `true`) is set via the `koa-sslify` middleware; flip it off when SSL terminates at an external load balancer. Auth-provider credentials (`SLACK_CLIENT_ID/SECRET`, `GOOGLE_CLIENT_ID/SECRET`, `AZURE_CLIENT_ID/SECRET`, `OIDC_*`) and integration credentials (`GITHUB_*`, `GITLAB_*`, `LINEAR_*`, `SLACK_VERIFICATION_TOKEN`, `FIGMA_*`, `NOTION_*`, `IFRAMELY_*`, `SENTRY_TUNNEL`) are not listed here — see `.env.sample` for the full set. File-storage size limits (`FILE_STORAGE_UPLOAD_MAX_SIZE`, `FILE_STORAGE_IMPORT_MAX_SIZE`, `FILE_STORAGE_WORKSPACE_IMPORT_MAX_SIZE`), the local-disk root (`FILE_STORAGE_LOCAL_ROOT_DIR`), optional `REDIS_COLLABORATION_URL` for horizontal collaboration scaling, and the `AWS_CLOUDFRONT_*` signed-URL triplet also live in `.env.sample`.

## Telemetry

When the master process boots, `checkUpdates()` (`server/utils/updates.ts`) sends an anonymous version ping to the maintainers. It contains the Outline version, deployment flavour (`cloud`/`community`/`enterprise`), and platform metadata — no document content, user identifiers, or IPs. Set `ENABLE_UPDATES=false` to opt out.

## Release process

A release is driven by `server/scripts/release.js`, invoked through `yarn release <major|minor|patch|x.y.z>`. The script prompts for confirmation, advances the version with `semver.inc`, rewrites `package.json#version`, updates `LICENSE` (Business Source License) with the new Licensed Work and Change Date (4 years out), commits both files as `v<ver>`, creates a `v<ver>` git tag, and pushes to origin. No CHANGELOG is generated; release notes are pulled from GitHub Releases.

The git tag triggers `.github/workflows/docker.yml`, which builds `Dockerfile.base` and `Dockerfile` for both `amd64` and `arm64` on Blacksmith runners, pushes each image with the tags `{{version}}` and `{{major}}.{{minor}}`, and merges them into a multi-arch manifest list with `docker buildx imagetools create`. Translation updates flow back from Crowdin through a separate bot workflow that commits files matching `fix: New %language% translations from Crowdin [ci skip]`. The full release pipeline (CI gates, Docker publish, Crowdin) lives in `docs/CI.md`.

### Upgrading an existing deployment

To bump dependencies in lockstep, use the standard `yarn upgrade` workflow (or edit a version in `package.json` and re-run `yarn install`). `yarn install-local-ssl` generates a self-signed cert pair into `server/config/certs/` for local HTTPS during development. To roll forward a self-hosted install to a new release tag, pull the new image (`outlinewiki/outline:vX.Y.Z` or your own build), stop the running process, start the new process with `--no-migrate` first if you intend to apply migrations separately, then start the regular process and let Umzug apply pending migrations on the master boot. Database backups and Redis snapshots should be taken before upgrading.

## File map

- `package.json` — build, test, lint, release, and db scripts.
- `build.js` — orchestrates the server-side Babel compile and the static copy step.
- `vite.config.ts` — bundle, PWA, vendor-chunk, sourcemap, and CDN-base configuration.
- `i18next-parser.config.js` — extraction rules that feed `yarn build:i18n`.
- `.babelrc` — Babel presets/plugins shared by build and tests; sets `SOURCE_COMMIT` and `SOURCE_VERSION`.
- `Dockerfile.base`, `Dockerfile` — builder and runner stages.
- `docker-compose.yml` — local Postgres and Redis for development.
- `Procfile`, `app.json` — Heroku process types and one-click deploy manifest.
- `.sequelizerc` — sequelize-cli configuration (loads `.env`/`.env.test` via dotenvx).
- `server/migrations/` — sequelize migration files.
- `server/scripts/release.js` — version bump, LICENSE patch, commit, tag, push.
- `server/utils/updates.ts` — anonymous version ping.
- `server/storage/database.ts` — Umzug auto-migrate wiring on master boot.
- `.env.sample` — full list of every environment variable recognised by the server.
- `.github/workflows/docker.yml` — tag-driven multi-arch image publish.
- `docs/DEVELOPMENT.md`, `docs/CI.md`, `docs/SERVICES.md` — adjacent processes.
